# Auto-Update System

MarketMind includes a robust auto-update system that automatically downloads and installs updates from GitHub releases, ensuring users always have the latest features and security fixes.

## Features

- **Automatic Background Checks**: Periodically check for updates without user intervention
- **GitHub Releases Integration**: Updates are published and distributed via GitHub releases
- **Download Progress Tracking**: Real-time progress indicators during download
- **Silent Installation**: Updates install seamlessly with minimal user disruption
- **User Control**: Configurable auto-check interval and download preferences
- **Native Notifications**: Desktop notifications when updates are available

## Architecture

### Components

1. **UpdateManager** (`src/main/services/UpdateManager.ts`)
   - Main process service managing electron-updater lifecycle
   - Handles autoUpdater events and configuration
   - Provides API for manual and automatic update checks

2. **IPC Handlers** (`src/main/index.ts`)
   - `update:check` - Manually check for updates
   - `update:download` - Download available update
   - `update:install` - Quit and install downloaded update
   - `update:getInfo` - Get current update information
   - `update:startAutoCheck` - Start periodic update checks
   - `update:stopAutoCheck` - Stop automatic checking

3. **UpdateAPI** (`src/main/preload.ts`)
   - Secure bridge exposing update methods to renderer process
   - Type-safe IPC communication

4. **useAutoUpdate Hook** (`src/renderer/hooks/useAutoUpdate.ts`)
   - React hook managing update state
   - Provides status, progress, and control methods

5. **UpdateNotification Component** (`src/renderer/components/Update/UpdateNotification.tsx`)
   - User interface for update notifications
   - Shows download progress and install prompts

6. **Settings Integration** (`src/renderer/components/Settings/GeneralTab.tsx`)
   - User preferences for auto-update behavior
   - Manual update check button

## Configuration

### User Settings

Users can configure auto-update behavior in Settings > General:

- **Check for updates automatically**: Enable/disable automatic update checks
- **Check interval**: How often to check for updates (1-168 hours)
- **Download updates automatically**: Auto-download when available
- **Check for Updates Now**: Manual update check button

Settings are persisted in localStorage:
- `autoCheckUpdates`: boolean (default: true)
- `updateCheckInterval`: number (default: 24 hours)
- `autoDownloadUpdates`: boolean (default: true)

### UpdateManager Configuration

```typescript
autoUpdater.autoDownload = false;  // Manual download control
autoUpdater.autoInstallOnAppQuit = false;  // Manual install control
autoUpdater.logger = log;  // electron-log integration
```

## Update Flow

### Automatic Updates

1. App starts → UpdateManager initializes
2. If `autoCheckUpdates` enabled → Start periodic checks (default: 24h)
3. Update found → Notify user via UpdateNotification
4. If `autoDownloadUpdates` enabled → Download automatically
5. Download complete → Prompt user to install
6. User clicks "Install and Restart" → App quits and installs

### Manual Updates

1. User opens Settings > General
2. User clicks "Check for Updates Now"
3. App checks GitHub releases
4. If update available → Show UpdateNotification
5. User clicks "Download" → Download starts with progress bar
6. Download complete → User clicks "Install and Restart"

## Publishing Updates

### GitHub Releases Workflow

1. **Update Version**
   ```bash
   npm version patch  # or minor/major
   ```

2. **Build Distributables**
   ```bash
   npm run build:mac    # macOS DMG
   npm run build:win    # Windows NSIS installer
   npm run build:linux  # Linux AppImage
   ```

3. **Create GitHub Release**
   ```bash
   git tag v0.11.0
   git push origin v0.11.0
   ```

4. **Upload Artifacts**
   - Upload files from `dist/` to GitHub release
   - Include: `.dmg`, `.exe`, `.AppImage`, and `latest*.yml` files
   - The `latest-mac.yml` and `latest.yml` files are auto-generated by electron-builder

### Release Naming Convention

- Tag: `v{major}.{minor}.{patch}` (e.g., `v0.11.0`)
- Draft releases are ignored by auto-updater
- Pre-releases can be opted into via configuration

### Required Files

electron-updater requires these files in each release:

**macOS:**
- `MarketMind-{version}.dmg`
- `latest-mac.yml`

**Windows:**
- `MarketMind-Setup-{version}.exe`
- `latest.yml`

**Linux:**
- `MarketMind-{version}.AppImage`
- `latest-linux.yml`

## Development & Testing

### Testing Auto-Update Locally

1. **Build a distributable**
   ```bash
   npm run build:mac  # or build:win
   ```

2. **Install the built app**
   - Install from `dist/MarketMind-{version}.dmg`

3. **Create a newer version**
   ```bash
   npm version patch
   npm run build:mac
   ```

4. **Create local GitHub release**
   - Create release with new version tag
   - Upload newer build artifacts

5. **Test update flow**
   - Open installed app (older version)
   - Click "Check for Updates Now"
   - Verify update detection and download

### Environment Variables

For development testing with a different update server:

```env
# .env
VITE_UPDATE_SERVER_URL=https://your-test-server.com
```

Then in `UpdateManager.ts`:
```typescript
if (process.env.VITE_UPDATE_SERVER_URL) {
  autoUpdater.setFeedURL({
    provider: 'generic',
    url: process.env.VITE_UPDATE_SERVER_URL
  });
}
```

### Debugging

Enable verbose logging in `UpdateManager.ts`:

```typescript
log.transports.file.level = 'debug';
autoUpdater.logger = log;
```

Logs location:
- **macOS**: `~/Library/Logs/marketmind/main.log`
- **Windows**: `%USERPROFILE%\AppData\Roaming\marketmind\logs\main.log`
- **Linux**: `~/.config/marketmind/logs/main.log`

## Update States

The auto-update system has the following states:

| State | Description | User Action |
|-------|-------------|-------------|
| `idle` | No update activity | Can check for updates |
| `checking` | Checking for updates | Wait for result |
| `available` | Update found | Download or dismiss |
| `downloading` | Downloading update | View progress |
| `downloaded` | Ready to install | Install and restart |
| `error` | Update failed | Dismiss and retry |

## Security Considerations

### Code Signing

Updates **must** be code-signed to ensure authenticity:

**macOS:**
```bash
export CSC_LINK=path/to/certificate.p12
export CSC_KEY_PASSWORD=certificate-password
npm run build:mac
```

**Windows:**
```bash
export CSC_LINK=path/to/certificate.pfx
export CSC_KEY_PASSWORD=certificate-password
npm run build:win
```

### Notarization (macOS)

macOS apps must be notarized for Gatekeeper:

1. Configure `scripts/notarize.js` with Apple credentials
2. Build with notarization:
   ```bash
   export APPLE_ID=your-apple-id@email.com
   export APPLE_APP_SPECIFIC_PASSWORD=app-specific-password
   npm run build:mac
   ```

See [BUILD.md](./BUILD.md) for complete code signing setup.

## Troubleshooting

### Update Not Detected

- Verify GitHub release is published (not draft)
- Check `latest-mac.yml` / `latest.yml` files are uploaded
- Ensure release tag follows `v{version}` format
- Check app version is older than release version

### Download Fails

- Verify GitHub release assets are public
- Check network connectivity
- Review logs for error details
- Ensure sufficient disk space

### Installation Fails

- Verify app is code-signed
- Check user has write permissions
- Review error logs
- Try manual installation from DMG/installer

### Auto-Check Not Working

- Verify `autoCheckUpdates` setting is enabled
- Check `updateCheckInterval` value
- Ensure app is running (not just in tray)
- Review logs for errors

## Best Practices

1. **Always increment version** before publishing updates
2. **Test updates** with local builds before public release
3. **Code-sign all releases** for security and trust
4. **Include release notes** in GitHub releases for transparency
5. **Monitor logs** for update-related errors
6. **Keep update intervals reasonable** (24h minimum recommended)
7. **Don't force immediate installation** - let users choose timing
8. **Test on all platforms** before public release
